# File: .ai/skill/AI_AGENT_RULES.md

# 1. Before Writing Code

### Checklist
- [ ] **Explore Codebase**: Read `AGENTS.md`, `README.md`, and relevant `.ai/skill/*.md` files to understand repository-specific constraints.
- [ ] **Verify Function Existence**: Do not assume the existence of specific functions, methods, or API endpoints. Confirm their presence via `grep` or `cat` during the exploration phase.
- [ ] **Check Dependencies**: Confirm you are working within the correct pnpm workspace (`packages/api`, `packages/web`, etc.) and note its existing dependencies in `package.json`.
- [ ] **Prioritize Consistency**: Prioritize consistency with existing code over idealized architecture. If a pattern cannot be verified from source code, state "Pattern not found in repository."

# 2. Before Editing Existing Code

### Checklist
- [ ] **Trace Artifacts to Source**: If editing a file in a build output directory (`dist`, `build`), trace the code back to its source and edit the source instead.
- [ ] **Check for Auto-Generated Files**: Ensure the file does not have a "Do not edit this file directly" guardrail (e.g., Prisma generated files).
- [ ] **Review Testing Implications**: Identify existing unit or integration tests that cover the code being modified and plan to update them accordingly.
- [ ] **Preserve Existing Documentation Rules**: When modifying `AGENTS.md` or `README.md`, preserve existing rules by nesting them under new headings rather than overwriting.

# 3. Before Creating New Files

### Checklist
- [ ] **Determine Package Boundary**: Ensure the new file is placed in the correct workspace package based on its domain (e.g., frontend code in `packages/web/src/`).
- [ ] **Check Test Directory Rules**: Tests must be placed in `tests/unit/` or `tests/integration/` directories within the package, not directly inside `src/`.
- [ ] **Verify Naming Conventions**: Use `camelCase` for variables/functions and `PascalCase` for classes/components/interfaces. Use `.ts` or `.tsx` appropriately.

# 4. Before Finishing Task

### Checklist
- [ ] **Clean Up Workspace**: Ensure no temporary scratchpad scripts (`.sh`, `.js`, `.py`) are left in the working directory.
- [ ] **Format Code**: Run `pnpm exec prettier --write <filepath>` or `pnpm run format` from the root.
- [ ] **Run Lint and Type Checks**: Run `pnpm run lint` and `pnpm run build` (which includes `tsc -b`).
- [ ] **Run Tests**: Execute `pnpm run ci` from the workspace root to ensure no regressions were introduced across the monorepo.
- [ ] **Format PR Title**: Ensure the commit/PR title matches project conventions (e.g., `🔒 [security fix description]`, `🧪 [testing improvement description]`).

# 5. Forbidden Actions

### Architecture Violations
- **Deep Cross-Package Imports**: Relying on relative paths to access other packages (e.g., `../../api/src/helpers/redactError`). Always import from a package's public exports or duplicate small utilities locally.
- **Introducing Unapproved Heavy Dependencies**: Adding heavy libraries or global state management tools without strict necessity.
- **Bypassing Abstractions**: Skipping ClickHouse parameter binding and using direct template strings for user input.

### TypeScript and Quality Violations
- **Unsafe Casting**: The use of `any` or unsafe casting (like `as any`) is strictly prohibited in application source code.
- **Hardcoded Secrets**: Providing fallback values for missing environment secrets (e.g., `process.env.JWT_SECRET || 'fallback'`).

### Process Violations
- **Modifying Output Directories**: Changing files in build/dist directories instead of the source files.
- **Inventing Conventions**: Recommending or implementing generic "best practices" that are not explicitly documented or actively used in the repository.
